# Represent by consequence

Spend simulation, precision, and storage only where a consumer depends on the result.

## Formal Expression

1. **Demand-shaped work.** A quantity is computed, refined, or stored because some consumer — a later phase, a neighbouring system, a query, an observer — depends on it, not because a uniform grid or a demo path is easier if everything is always dense.
2. **Three axes of the same principle.** The principle runs along (a) multiscale substrate machinery, (b) developmental phase order (systems come online when a Charge needs them), and (c) use-case as fidelity contract (what the finished world is *for* sets which axes must be honest). Detailed elaboration lives in source design prose until those axes have their own segments.
3. **Fidelity invariant (statement).** Render and simulate at the resolution — spatial and temporal — the most demanding present participant needs; do not invent finer truth that nothing can use and nothing can check.

## Epistemic Status

**Max attainable: axiomatic** as project stance. Stated as the consolidating principle of `doc/ARCHITECTURE.md`. Stage `draft`. Sub-claims (lazy memo graph, phase freezes, consumer-dependent restriction statistics) need their own segments before they inherit this postulate's authority; this file does not smuggle them in as proved.

## Discussion

The opposite failure is familiar: compute everything at finest grain "to be safe," then neither finish the world nor know which numbers were load-bearing. Represent-by-consequence is how a planet-scale vivium stays both playable and citable.
